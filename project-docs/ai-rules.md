# Правила проекта

## Что мы делаем
Сайт регулярно пополняется новыми статьями по контент-плану.

## Правила для AI
- Использовать контент-план
- Сохранять стиль проекта
- Не менять старые статьи
- Не придумывать факты

## Стиль
### Общие принципы
- Публицистичный, ясный, сдержанный язык.
- Текст для интеллектуальной аудитории без углубленных знаний . фотографии.
- Приоритет смысла над риторикой: каждая фраза должна нести информацию.
- Логика изложения последовательная: тезис → пояснение → пример → вывод (внутри абзаца или раздела).

### Формулировки
- Короткие и средние предложения; избегать громоздких конструкций.
- Конкретные глаголы вместо абстракций ("показывает", "смещает внимание", "усиливает контраст").
- Минимум вводных слов и усилителей ("очень", "довольно", "в целом").
- Термины использовать точно и единообразно по всей статье.
- Избегать близких повторов одного и того же слова в соседних предложениях или внутри одного абзаца; при необходимости заменять повтор синонимом ("снимок", "кадр", "фотография", "изображение") или перестраивать фразу.

### Абзацы и ритм
- Короткие абзацы (2–5 предложений).
- Один абзац ─ одна мысль.
- Плавные переходы между абзацами без «скачков».
- Всегда тексте использовать "─", никогда не используй "длинные тире"
- Первый абзац статьи должен быть лидом: короткий вход в тему с главным тезисом или проблемой, который задаёт смысл чтения.

### Логика и структура
- Каждый раздел решает конкретную задачу, заявленную в плане.
- Нет повторов одних и тех же тезисов в разных формулировках.
- Примеры (включая иллюстрации) напрямую поддерживают тезис раздела.
- Введение открывается лидом; после лида текст сразу переходит к постановке задачи статьи без разгона.

### Тон
- Уважительный, нейтральный, без фамильярности.
- Без поучительного или назидательного звучания.
- Без обращений к читателю в повелительном наклонении, если это не требуется смыслом.

### Что строго избегать
- Конструкций вида «это не просто …».
- Конструкций вида «не только …, но и …».
- Риторических украшений, метафор и образных сравнений.
- Сленга и разговорных слов.
- Пустых вступлений (общие рассуждения без конкретного тезиса).
- Банальных выводов (повтор содержания без добавления смысла).
- Воды: фраз, которые можно удалить без потери смысла.

### Дополнительные ограничения
- Не переводить устоявшиеся термины, если в профессиональной среде используется оригинал.
- Имена авторов и названия работ ─ на языке оригинала.
- Не использовать клише и шаблонные обороты.

### Критерий качества

Если из абзаца можно удалить 20–30% текста без потери смысла ─ абзац требует уплотнения.

## Иллюстрации в статьях

0. При создании черновика новой статьи сразу создавать папку для будущих иллюстраций:
- путь: `assets/images/slug-статьи`
- если изображений ещё нет, добавлять в папку `.gitkeep`, чтобы папка сохранялась в репозитории
- папку создавать одновременно с файлом черновика, без отдельной команды пользователя

1. Вставлять иллюстрации только по команде пользователя:
   «вставь изображение» / «добавь иллюстрацию»

2. Всегда использовать формат:
[Иллюстрация]
Автор ─ Название фотографии
Почему здесь, текст в стиле статьи, созданный по следующим правилам:
- какой тезис подтверждает
- что именно показывает

Допустимое специальное исключение для первого дескриптора в статье:
[Обложка статьи]
Почему здесь:
- краткий текст, который должен перейти в `alt` обложки при замене дескриптора шорткодом

3. Использовать только конкретные фотографии:
- запрещены общие описания («портрет», «улица»)
- обязательно: конкретный автор + конкретное название работы
- конкретная фотография может использоваться в статье только один раз
- конкретная фотография не может повторяться чаще одного раза за 2 месяца публикационного плана или в пределах ближайших 25 статей
- при подборе любой иллюстрации всегда сначала сверяться с реестром ограничений в `project-docs/used-illustrations.md`; без этой проверки иллюстрацию не вставлять
- проверку реестра выполнять перед каждой отдельной иллюстрацией, а не один раз в начале работы над статьёй
- это правило относится и к замене уже вставленной иллюстрации: перед каждой заменой снова проверять `project-docs/used-illustrations.md`
- если фотография вставлена, сразу добавлять её в `project-docs/used-illustrations.md`
- если фотография уже есть в активном ограничении, выбирать другую работу
- `content/blog` проверять как дополнительную страховку, если есть сомнение, что реестр ещё не содержит старую иллюстрацию
- если фотография уже встречалась недавно, выбирать другую работу, даже если тезису она подходит

4. Источник авторов:
- преимущественно использовать список:
https://school.gurfoto.ru/invisibility/
- не добавлять авторов вне списка без запроса
- не искать иллюстрации в текущем проекте

5. Обязательное объяснение:
- всегда писать блок «Почему здесь»
- связывать изображение с тезисом статьи
- избегать общих фраз

6. Количество:
- по умолчанию: 1 иллюстрация на смысловой блок
- не вставлять изображения без запроса

7. Команда «Замени дескриптор шорткодом»:
- команда означает заменить дескриптор иллюстрации на Hugo-шорткод `imgr`
- обрабатывать только один дескриптор, указанный пользователем; если дескриптор не указан явно, уточнить у пользователя, какой именно блок заменить
- вместо строки `[Иллюстрация]` вставлять шорткод:
`{{< imgr src="путь/к/файлу" alt="Автор ─ Название фотографии" >}}`
- если первый дескриптор в тексте имеет вид `[Обложка статьи]`, то он считается специальным случаем:
- на его место вставлять шорткод с файлом `... 1.*` из папки статьи внутри `assets/images`
- путь в `src` указывать относительно `assets/images`
- текст после `Почему здесь:` переносить в параметр `alt` шорткода
- подпись под шорткодом не оставлять
- строку `Почему здесь:` удалять из статьи
- текст после `Почему здесь:` удалять из статьи после переноса в `alt`
- путь в `src` указывать относительно `assets/images`: `папка-статьи/имя-файла`
- графические файлы лежат в папке статьи внутри `assets/images`; номер в имени файла соответствует положению иллюстрации в тексте черновика
- если после frontmatter уже стоит обложка с файлом `... 1.*`, номер файла для дескриптора считать по его положению среди иллюстраций в тексте
- перед заменой проверять, что файл существует
- строку `Автор ─ Название фотографии` копировать в параметр `alt` без изменений
- строку `Автор ─ Название фотографии` в тексте оставлять под шорткодом и оборачивать в `*` с двух сторон
- строку `Почему здесь:` удалять; между подписью и поясняющим текстом оставлять пустую строку
- поясняющий текст после `Почему здесь:` сохранять без изменения, если пользователь не просит редактировать его

## Стилистическая обработка текста

Цель: привести черновик к ясному, точному и ровному стилю без изменения смысла.

Правила:
- упрощать длинные и перегруженные предложения
- убирать повторы и дублирующие мысли
- сохранять единый спокойный тон
- заменять общие формулировки на конкретные
- устранять логические скачки и резкие переходы
- не добавлять новую информацию и факты

Порядок действий (всегда по шагам):
1. Упростить язык
2. Убрать повторы
3. Выровнять стиль
4. Усилить точность формулировок
5. Проверить логику и переходы

Команды:
- «сделай текст проще и яснее»
- «убери повторы, сохрани смысл»
- «выровняй стиль»
- «сделай формулировки более точными»
- «проверь логику и переходы»
- «сделай полную стилистическую обработку черновика»
